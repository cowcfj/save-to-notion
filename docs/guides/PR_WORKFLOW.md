# 🔄 PR 與代碼審核工作流程

> **核心精神**：保持單一職責、嚴格把控日誌安全、確保覆蓋率不下降。

---

## 📋 提交與分支規範

### 分支命名

- `feature/*` - 新功能
- `fix/*` - 錯誤修復
- `chore/*` - 構建、工具、依賴
- `docs/*` - 文檔更新
- `refactor/*` - 代碼重構

### 提交訊息 (Conventional Commits)

格式：`類型: 簡短描述` (例如：`feat: 新增可搜索資料庫選擇器` 或 `fix: 修復標註在長文章中的恢復問題`)

## 🛡️ 合併前安全與質量自檢指引 (Pre-flight Guidelines)

> `.github/pull_request_template.md` 是你最終填寫的**表單**。
> 本節則是解釋**為什麼**你要勾選那些項目，以及背後的審核標準。

### 1. 日誌安全性（🚨 絕對紅線）

為什麼模板會要求檢查 `sender` 或 `tab`？

- **無完整物件記錄**：擴充功能的 Background Script / Content Script 中，禁止將充滿敏感資訊的完整物件 (如 `sender`, `tab`, `request`) 直接印出至 `console`。你只能記錄其 ID (`senderId`, `tabId`)。
- **清理外部輸入**：外部 URL 需經過 `sanitizeUrlForLogging()` 處理以防 XSS 或 PII 洩漏；API 錯誤訊息需經過 `sanitizeApiError()`。

### 2. 架構與防禦性設計

- **單一主題 (Cohesion)**：這個 PR 的變更是否足夠內聚？如果修 Bug 的同時夾帶了不相關的重構，Reviewer 可能會要求你拆分 PR。
- **防呆與向後兼容**：是否有考慮到舊數據的兼容？用戶可能運行著舊版的 Storage Schema，輸入是否已被適當驗證 (`isValidUrl` 等)？

## 📝 PR 結構與模板 (Templates & Machine Instructions)

為確保無論是**人類手動開發**還是由 **AI 自動化執行**，都能產出一致且符合規範的 Pull Request，我們對 PR Content 有嚴格要求。

### 1. 👨‍💻 人工提交 PR（VS Code / Terminal）

當你在 VS Code 透過 Git 擴充功能（如 GitHub Pull Requests and Issues），或在終端機使用 `gh pr create` 時：

- 系統應會自動帶入 `.github/pull_request_template.md` 的內容。
- 若工具未自動抓取，請**切勿手動發明格式**。請複製該模板內容貼入 PR Body 中。
- 請詳實地將模板中對應的 Checkbox `[ ]` 改為 `[x]`。

### 2. 🤖 AI Agent 自動 PR 強制設定 (Machine-Readable Instructions)

> **CRITICAL INSTRUCTION FOR ALL AI AGENTS:**
> When you are instructed to create, update, or propose a Pull Request using GitHub API or MCP tools:
>
> 1. You **MUST NOT** generate your own PR description format.
> 2. You **MUST** read the file `.github/pull_request_template.md` FIRST.
> 3. Your entire PR body (`body` parameter in API request) **MUST** strictly use this exact template format.
> 4. You must actively evaluate the changes you made and replace `[ ]` with `[x]` ONLY for the items you have verifiably completed.
> 5. Clearly state at the top or bottom of the PR whether this was generated by an AI Agent.

## ⚙️ 儲存庫分支保護政策 (Branch Protection)

為防止意外破壞主線，GitHub 儲存庫已實施以下保護：

- **禁止直接 Push**：必須透過 PR 合併至 `main` 或 `develop`。
- **Status Checks Required**：自動化構建與預設的 CI 檢查若失敗，將阻擋 Merge 按鈕 (Blocker)。
- **Conversation Resolution**：所有 Code Review 的討論串必須被清楚標示為 `Resolved` 才能合併。
