| **0️⃣ GOALS.md**           | AI Agent、開發團隊 | 每季度/大版本  | **戰略目標**。專案路線圖與優先級清單。**不要放入實作細節**，當功能從「規劃中」轉為「開發中」時，交由 `writing-plans` 技能產出實作計畫。  |

| **5️⃣ help.html**          | 擴展用戶 (內嵌)    | UI/UX 重大更新 | **快速導流**。極簡設計（<200行程式碼），僅保留 Top 5 FAQ 與快速開始，詳情全部導流至 `USER_GUIDE.md`。                                    |

## 🔄 發布與更新檢查清單 (Release Checklist)

無論是大版本或小版本更新，在準備發布時，請確認以下流程：

- [ ] **版本號一致性**：同步更新 `manifest.json` 與 `package.json` 中的版本號。
- [ ] **檢查自動發布 PR**：確認 `release-please` 自動產生的發布 PR (Release PR) 內容是否正確。
- [ ] **更新 README.md**：大版本發布時，需將上一版的詳情內容化為單行摘要，並確保「最新更新」區塊正確無誤。
- [ ] **確認 CHANGELOG.md**：(由 Action 產生，僅需 Review) 確保分類清晰 (Added, Fixed 等)，有必要時調整提交紀錄語氣。
- [ ] **同步指南 (若需)**：檢視是否有新功能或 UI 變更，需補充至 `USER_GUIDE.md` 操作步驟，或調整 `help.html` 中的快捷鍵/FAQ。

## 🏗️ 技術規格與架構文檔策略 (docs/specs/)

對於 `docs/specs/` 目錄下的技術規格與架構白皮書，我們遵循以下生命週期與撰寫規範：

### 1. 文檔類型

- **核心架構白皮書 (Core Architecture)**：專案的最高技術指導原則 (Single Source of Truth)，解釋系統架構、安全性哲學與核心設計決策。為長期維護、與程式碼高度同步的 Living Document。
- **功能規格 (Specifications)**：描述計劃實現的功能，包含需求、設計、實施計劃。
- **研究與評估 (Research & Evaluation)**：技術調研、可行性分析、工具比較與決策參考。

### 2. 文檔狀態標記

建議在規格文檔開頭標記狀態：

- **狀態**: 📋 規劃中
- **狀態**: 🔄 實施中
- **狀態**: ✅ 已實現

### 3. 文檔生命週期

1. **規劃階段**：創建規格文檔於 `docs/specs/future/`，狀態為 `📋 規劃中`。
2. **開始實施**：移至 `docs/specs/` 根目錄，狀態改為 `🔄 實施中`。
3. **完成階段**：
   - 若為功能或重構：將核心架構決策與經驗提煉 (Extract) 至**核心架構白皮書**中。提煉完成後，將該任務文件移動到 `docs/specs/archived/` (或直接刪除以保持精簡)。
   - 若為長期獨立功能，更新狀態為 `✅ 已實現`，並保留在 `docs/specs/`。
4. **歸檔階段**：功能已完成且不再需要獨立維護，或已棄用時，移至 `docs/specs/archived/`。

### 4. 創建新文檔的指南

1. **選擇合適的文件名**：使用大寫字母與下劃線，並加上明確的類型後綴 (如 `_SPEC.md`, `_ARCHITECTURE.md`)。
2. **包含必要的元數據**：在文件開頭標明狀態、創建日期、作者等。
3. **使用清晰的結構**：內容應包含問題/背景、目標/需求、方案/設計、實施步驟等。

## ⚖️ 架構決策紀錄策略 (docs/adr/)

架構決策紀錄 (Architecture Decision Records, ADR) 用於捕捉並永久保存專案中「重大且難以逆轉」的技術決策過程。

### 1. 何時該寫 ADR？

以下情況**必須**建立 ADR：

- 引入新的重大第三方依賴庫、框架或 AI 技能。
- 改變資料儲存結構、狀態同步方式 (如：從 `chrome.storage.sync` 改為 `local` 並掛載 IndexedDB)。
- 權衡兩難的介面設計或 API 通訊協議 (如：Service Worker 內部直接 fetch 還是 Message Passing)。
- **任何在 Code Review 時會被問「為什麼不那樣做？」的決策。**

### 2. ADR 的核心原則

- **不變性 (Immutable)**：一旦 ADR 被標記為 `Accepted`，就**永遠不應該被修改**。
- **上下文比結果更重要**：未來的開發者（或 AI）最需要知道的是「當時面臨什麼限制」，而不是單純的「我們選了 A」。
- **決策覆寫**：如果未來的架構改變了，不要去修改舊的 ADR。你應該**建立一份新的 ADR**，並在狀態中標記它「Supersedes ADR-XXXX」，同時將舊的 ADR 狀態改為「Superseded by ADR-YYYY」。

### 3. 如何建立 ADR

1. **複製模板**：複製 `docs/adr/0000-template.md`。
2. **命名規範**：使用 `四位數流水號-小寫英文-kebab-case.md`，例如 `0002-migrate-to-indexeddb.md`。
3. **填寫內容**：確保 `Context` 與 `Consequences`（無論好壞）被誠實且詳盡地記錄。
